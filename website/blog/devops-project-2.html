Setting Up Nexus Repository Manager: When Things Don't Start (And What To Do About It)

February 11, 2026
10 min read
Noah Vastola
Nexus DigitalOcean Linux Java Gradle Maven

Run Nexus on Droplet and Publish Artifact to Nexus

After getting my first Java application running on DigitalOcean, the next project seemed straightforward: install Nexus Repository Manager on another server and use it to store build artifacts. Nexus is basically a private storage system for JAR files, Docker images, npm packages—anything your team builds that you want to store centrally.

The idea makes sense. Instead of developers building artifacts locally and passing them around, you build once, store it in Nexus, and everyone pulls from the same source. Version control for your compiled code.

I followed the installation guide, downloaded Nexus, set up the directories, configured the service, and ran the start command.

Nothing happened.

The Silent Failure

When I checked if Nexus was running with systemctl status nexus, it said "active (running)." Great! Except when I tried to access it at http://<server-ip>:8081, I got nothing. Connection refused.

I checked if anything was listening on port 8081:

sudo netstat -tulpn | grep 8081

Nothing. No process listening on that port. But the service status said it was running. How could it be running if nothing was listening?

I checked the logs in /opt/sonatype-work/nexus3/log/ but they were... empty. No errors. No warnings. Just silence.

The Worst Kind of Error

This is the worst kind of error: the silent failure. No logs, no error messages, no indication of what's wrong. Just a service that claims to be running but isn't actually doing anything.

Running Nexus in the Foreground

After some frustrated Googling, I found a suggestion to run Nexus in foreground mode instead of as a background service. This would force it to output logs directly to the terminal: 

/opt/nexus/bin/nexus run

The moment I ran this, I got actual output. And buried in that output was an error:

/opt/nexus/bin/nexus: /opt/nexus/bin/nexus: cannot execute binary file: Exec format error

Exec format error. That's... not something you see every day.

I stared at this for a moment before it clicked: I must have downloaded the wrong version of Nexus.

The Architecture Mismatch

I ran uname -m to check my server's architecture:

uname -m

Output: x86_64

Then I checked what I had downloaded. The filename was something like nexus-3.x.x-unix-aarch64.tar.gz.

Architecture Mismatch

There it was. aarch64. That's ARM architecture. My server was x86_64 (Intel/AMD architecture). I had downloaded the ARM version of Nexus and tried to run it on an x86 server. That's like trying to run a Mac app on a Windows machine—the operating system might be similar, but the processor architecture is completely different.

I deleted everything, downloaded the correct unix tarball (which is x86_64), extracted it, and tried again.

This time when I ran /opt/nexus/bin/nexus run, I actually saw Nexus starting up. Logs scrolled by. Services initialized. And finally: 

Started Sonatype Nexus OSS

I accessed http://<server-ip>:8081 in my browser, and there it was: the Nexus login page.

Success!

Lesson learned: Always verify your server's architecture before downloading binaries. And when something fails silently, try running it in foreground mode to see the actual error output.

Setting Up Nexus for Real Use

With Nexus finally running, I logged in with the default admin credentials and immediately changed the admin password (security first).

The next step was creating a dedicated user for uploading artifacts. In a real company, you wouldn't give developers the admin credentials to Nexus. You'd create users with specific permissions.

I navigated to Security → Users and created a new user. Let's call them ci-user. Then I created a custom role for this user.

Understanding Nexus Permissions

Nexus has a pretty granular permission system. For every action you can perform (browse, read, edit, delete, add), there's a specific privilege. And for every repository, you can assign different privileges.

I noticed two main categories of permissions:

  • Repository Admin - For operations teams managing Nexus itself
  • Repository View - For developers uploading and downloading artifacts

Since my user just needs to publish and retrieve artifacts, I went with the "View" permissions. I gave the role these privileges:

  • nx-repository-view-maven2-maven-snapshots-* (all actions on Maven snapshots)
  • nx-repository-view-maven2-maven-releases-* (all actions on Maven releases)

I assigned this role to ci-user, and now I had a non-admin account ready for publishing artifacts.

Publishing from a Gradle Project

Nexus was running. The user was set up. Now I needed to actually use it.

The project I was working with used Gradle, so the first step was configuring Gradle to talk to Nexus. This required three things:

  1. A plugin to enable publishing
  2. Publication configuration (what to publish)
  3. Repository configuration (where to publish)

Adding the Maven Publish Plugin

In build.gradle, I added:

plugins {
    id 'maven-publish'
}

This plugin gives Gradle the ability to publish to Maven-format repositories. Even though this is a Gradle project, both Gradle and Maven use the same artifact format for Java libraries.

Configuring What to Publish

Next, I defined what artifact I wanted to publish:

publishing {
    publications {
        maven(MavenPublication) {
            artifact("build/libs/myapp-${version}.jar") {
                extension 'jar'
            }
        }
    }
}

This tells Gradle: "When you build, take the JAR file from build/libs/ and prepare it for publishing."

Configuring Where to Publish

Now for the repository configuration—this is where I point Gradle to my Nexus server:

publishing {
    repositories {
        maven {
            name 'nexus'
            url "http://<nexus-ip>:8081/repository/maven-snapshots/"
            credentials {
                username project.findProperty("repoUser") ?: "default"
                password project.findProperty("repoPassword") ?: "default"
            }
            allowInsecureProtocol = true
        }
    }
}

A few things to note here:

  • The URL structure: http://ip:port/repository/repository-name
    You can find the exact URL for each repository in Nexus by clicking on the repository and looking at the URL field.
  • The credentials: I'm NOT hardcoding the username and password in build.gradle. That file gets checked into Git, and you never want credentials in version control.

Instead, I created a gradle.properties file:

repoUser=ci-user
repoPassword=<password>

And I added gradle.properties to .gitignore immediately.

Security Note

allowInsecureProtocol: Since I'm using HTTP instead of HTTPS, Gradle requires this explicit flag. In production, you'd use HTTPS and wouldn't need this line.

Publishing the Artifact

With everything configured, I built the project:

./gradle build

This created build/libs/myapp-1.0-SNAPSHOT.jar.

Then I published it to Nexus:

./gradle publish

I watched as Gradle authenticated with Nexus, uploaded the JAR file, and... success!

I went back to Nexus, clicked Browse → Components, selected maven-snapshots, and there it was: 

com.example
└── myapp
    └── 1.0-SNAPSHOT
        ├── myapp-1.0-SNAPSHOT.jar
        ├── myapp-1.0-SNAPSHOT.pom
        └── maven-metadata.xml

My first artifact in a real artifact repository.

Publishing from a Maven Project

The process for Maven projects is similar but with different configuration files.

Configuring the Maven Plugin

In pom.xml, I added the maven-deploy-plugin:

<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-deploy-plugin</artifactId>
            <version>2.8.2</version>
        </plugin>
    </plugins>
</build>

Setting the Repository URL

Also in pom.xml, I added the distribution management section:

<distributionManagement>
    <snapshotRepository>
        <id>nexus-snapshots</id>
        <url>http://<nexus-ip>:8081/repository/maven-snapshots/</url>
    </snapshotRepository>
</distributionManagement>
Important

The <id> is important—it's how Maven knows which credentials to use.

Configuring Credentials

Maven stores credentials in a global settings file, not in the project. I created ~/.m2/settings.xml: 

<settings>
    <servers>
        <server>
            <id>nexus-snapshots</id>
            <username>ci-user</username>
            <password><password></password>
        </server>
    </servers>
</settings>

The <id> here matches the <id> in pom.xml. That's how Maven connects the dots: "Oh, this repository uses nexus-snapshots credentials."

Building and Publishing

mvn package  # Build the JAR
mvn deploy   # Upload to Nexus

And just like that, the Maven project's artifact appeared in Nexus alongside the Gradle one.

The Blob Store Cleanup Surprise

A few days later, I was checking disk space on my server and noticed Nexus was using way more space than expected. I had deleted some old artifacts from the UI, but the disk space hadn't decreased.

Deletion Isn't Immediate

Turns out, Nexus doesn't actually delete artifacts when you delete them from the UI. It marks them as deleted (a "soft delete"), but the files remain on disk.

To actually free up space, you need to run two tasks:

  1. Cleanup task - Marks artifacts for deletion based on your rules
  2. Compact blob store - Actually removes the marked artifacts from disk

I set up a cleanup task in Settings → Tasks to run nightly, and then manually ran the blob store compaction.

After the compaction ran, I checked disk usage again:

du -sh /opt/sonatype-work/nexus3/blobs/

The size had dropped significantly. The UI storage metrics didn't update immediately, so I had to run System → Tasks → Repair - Recalculate blob store storage to refresh the numbers.

Key Takeaway

Deleting artifacts in Nexus is a two-step process, and the UI doesn't always show accurate disk usage right away.

What I Learned

This project taught me several things that weren't obvious from the documentation:

  • Silent failures are the worst. When a service claims to be running but isn't, run it in foreground mode to see the actual error output.
  • Architecture matters. Download the right binary for your CPU architecture. ARM ≠ x86. Always verify with uname -m.
  • Credentials don't belong in version control. Use gradle.properties for Gradle projects and ~/.m2/settings.xml for Maven projects. Add these files to .gitignore.
  • Nexus cleanup is two-phase. Deleting artifacts in the UI doesn't free disk space immediately. You need both a cleanup task and blob store compaction.
  • Permission design matters. Don't give everyone admin access. Create roles with the minimum permissions needed for each use case.
  • Maven and Gradle use the same artifact format. Both publish to Maven-format repositories, so you can use one Nexus repository for both build tools.

Next Steps

With Nexus running and artifacts being published, the next phase of the bootcamp is containerization. I'll be learning Docker, building container images, and eventually deploying these same applications using Docker Compose on AWS.

I'm expecting more version conflicts, more networking issues, and probably some new failure modes I haven't encountered yet. But that's the whole point—every error teaches you how these systems actually work under the hood.

If you're working on something similar and run into issues, feel free to reach out. I'm documenting everything as I go, mistakes and all.
